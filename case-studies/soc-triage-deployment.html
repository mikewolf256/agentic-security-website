The Company

  • Industry: Fintech (consumer lending)
  • Stage: Series A, $5M ARR, 30 employees
  • Challenge: Preparing for SOC2, no security hire yet
  • Tech Stack: React/Node.js, WAF + observability tooling

The Problem

Their engineering team was drowning in security alerts:

  • 100+ alerts/day from WAF and observability alerts
  • 80% were false positives (legitimate users triggering rate limits)
  • Engineering spent 2-3 hours/day manually triaging Slack alerts
  • SOC2 auditor wanted proof of incident detection and response
  • Real threats buried in noise with no way to prioritize what mattered

Specific Pain Point: IDOR False Positives

Traditional rate limiting flagged any user accessing 5+ loan applications as "potential IDOR attack." But many legitimate customers had 10-15 active loans. Result: 50-70 false positive alerts daily, all requiring manual investigation.

The Solution

We deployed AI-powered triage middleware with ownership-aware detection:

Week 1: Deploy & Baseline

  • Deployed lightweight middleware as K8s sidecar in their VPC
  • Connected webhooks from existing alert sources (WAF + monitoring)
  • Added optional ownership context from application (who owns which loans)
  • Ran in shadow mode to establish baseline metrics

Week 2: Tune & Launch

  • Configured PII scrubbing for GDPR compliance
  • Added business context rules (QA users, test namespaces)
  • Enabled ownership-aware IDOR detection
  • Tuned thresholds: 3+ OTHER-user resources = escalate, own resources = ignore

Real Incident: The One That Mattered

In the first 30 days, the system caught 1 real IDOR attack that would have been buried in the noise before:

Attack Details

  • Pattern: User accessed 3 loan applications owned by different users
  • Time window: 45 seconds (sequential enumeration)
  • Verdict: CRITICAL_IDOR_ATTACK (95% confidence)
  • Response: Auto-hold triggered, account suspended, SOAR case created
  • Detection time: 19.6 seconds from first attempt

Why Traditional Systems Missed It

Before ownership-aware detection, this attack would have been flagged as:

  • "User accessed 3 resources" → Mixed in with 50+ daily "user accessed N resources" alerts
  • No ownership context → Looked identical to legitimate users with multiple loans
  • Engineering would have triaged eventually, but attacker had 45 seconds of access

With Ownership-Aware Detection

  • ✓ System recognized user does NOT own those 3 loans
  • ✓ Sequential pattern (loan_12345 → 12346 → 12347) indicated automation
  • ✓ High confidence verdict triggered auto-hold
  • ✓ SOAR case created with complete evidence
  • ✓ Engineering received 1 validated alert, not 50+ noise

Key Takeaways

  • Ownership context is the game-changer: Traditional rate limiting can't distinguish "user accessed N resources" from "user accessed N OTHER users' resources."
  • Business context reduces false positives: QA tags, deployment events, test namespaces prevent alert fatigue.
  • PII scrubbing enables AI adoption: GDPR/PCI-DSS compliance was non-negotiable. Local scrubbing solved it.
  • Audit trail is the deliverable: For SOC2, the logs and metrics matter as much as the detection itself.
