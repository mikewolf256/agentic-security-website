Validation in action: when "Critical" isn't real

This example shows how a common scanner heuristic can produce "critical auth bypass" findings, and how a lightweight validation loop can prove exploitability (or quickly close it out).

What the scanner flagged

NoSQL injection auth bypass
Scanner: Critical (CVSS 9.1)
Heuristic: 200 OK with homepage content instead of login form
LDAP injection auth bypass (3 variants)
Scanner: Critical (CVSS 9.1)
Heuristic: 200 OK with homepage content

How we validated it

  • Controls: baseline requests and empty/neutral payloads
  • Payload variants: multiple payloads per class (NoSQL/LDAP) to rule out one-off behavior
  • Response comparison: status, size, and content markers
  • Behavior check: does this endpoint actually perform authentication logic?

What we found

The root endpoint returned the same homepage response for all requests (payload and control), indicating the response-shape "signal" was an artifact, not exploitability.

Example outcome (simplified):
- Normal request   -> 200, ~400KB homepage
- Payload request  -> 200, ~400KB homepage (same markers)
- Empty request    -> 200, ~400KB homepage (same markers)
Conclusion: endpoint returns homepage regardless of input; no auth logic exercised.
Why this matters: Without validation, teams waste days on "critical" findings that aren't real. With validation, the deliverable is trustworthy and immediately actionable.

What you should expect in our deliverables

  • Status: confirmed / unconfirmed / false positive
  • Evidence: request/response excerpts and rationale
  • Reproduction: steps you can re-run immediately
